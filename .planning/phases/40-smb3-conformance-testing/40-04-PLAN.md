  <action>
Create `test/e2e/cross_protocol_lease_test.go` with `//go:build e2e` tag, package `e2e`.

Follow the dual-protocol server setup pattern from cross_protocol_test.go: start DittoFS with both NFS and SMB adapters on the same shared metadata/payload stores.

**Test implementations:**

1. **TestCrossProtocol_SMBLeaseBreakOnNFSWrite**:
   - Set up dual-protocol server (NFS + SMB, shared stores)
   - Connect via go-smb2, open file with read access (go-smb2 requests lease automatically)
   - Write to the same file via NFS mount
   - Wait briefly (500ms) for lease break propagation
   - Read file via SMB, verify content matches NFS-written data
   - This validates: SMB lease break fires on NFS write, data consistency maintained

2. **TestCrossProtocol_NFSDelegationRecallOnSMBOpen**:
   - Open file via NFS mount (read delegation may be granted)
   - Open same file via SMB for write
   - Write via SMB
   - Read via NFS, verify content matches SMB-written data
   - This validates: NFS delegation recall fires on SMB open

3. **TestCrossProtocol_SMBDirLeaseBreakOnNFSCreate/Delete/Rename**:
   - Connect via go-smb2, open directory (triggers directory lease)
   - Perform NFS operation (create file / delete file / rename file in that directory)
   - Wait briefly for break propagation
   - ReadDir via SMB, verify new state visible
   - Three separate subtests for create, delete, rename

4. **TestCrossProtocol_ConcurrentLeaseConflicts**:
   - Create 10 files
   - Launch 5 goroutines: each opens a random file via SMB, reads via NFS, writes via NFS, reads via SMB
   - Launch 5 goroutines: each opens a random file via NFS, reads via SMB, writes via SMB, reads via NFS
   - Use sync.WaitGroup for coordination
   - Verify no panics, no deadlocks, all operations complete within 30s
   - Verify final file contents are consistent (last writer wins)

5. **TestCrossProtocol_DataConsistencyAfterBreak**:
   - SMB client opens file, writes "version1"
   - NFS client writes "version2" to same file (triggers SMB lease break)
   - SMB client re-reads file, must see "version2"
   - SMB client writes "version3"
   - NFS client reads file, must see "version3"
   - This is the definitive data consistency validation

Use `time.Sleep(500ms)` for cross-protocol metadata sync (existing pattern from cross_protocol_test.go). Use `cache=none` mount option for SMB if using mount.cifs. For NFS, use `actimeo=0`.

IMPORTANT: For SMB operations, prefer go-smb2 (from plan 03 helpers) over smbclient for programmatic control. For NFS operations, use file system operations on the NFS mount point.

**Test implementations:**

1. **TestSMB3_Kerberos_SessionSetup**:
   - Set up KDC container via framework.NewKDCHelper
   - Create Kerberos principal for test user
   - Configure DittoFS with Kerberos keytab
   - Establish SMB3 session using Kerberos/SPNEGO auth (smbclient --use-kerberos=required or kinit + go-smb2)
   - Verify session established successfully

2. **TestSMB3_Kerberos_FileOps**:
   - Authenticate with Kerberos
   - Create file, write content, read back, verify
   - Create directory, list contents
   - Rename file, delete file

3. **TestSMB3_Kerberos_WithEncryption**:
   - Configure share with EncryptData=true
   - Authenticate with Kerberos
   - Perform file write/read
   - Verify operations succeed (encryption transparent to client)

4. **TestSMB3_Kerberos_WithSigning**:
   - Configure adapter with signing required
   - Authenticate with Kerberos
   - Perform file write/read
   - Verify operations succeed

5. **TestSMB3_Kerberos_NTLMFallback**:
   - Do NOT set up KDC or configure Kerberos keytab
   - Attempt connection with NTLM credentials
   - Verify NTLM fallback works and file operations succeed

6. **TestSMB3_Kerberos_GuestSession**:
   - Connect as guest (anonymous or with guest flag)
   - Verify basic operations work without encryption/signing
   - Verify guest has appropriate (limited) permissions

7. **TestSMB3_Kerberos_CrossProtocol**:
   - Set up dual-protocol server with Kerberos configured for both NFS (RPCSEC_GSS) and SMB (SPNEGO)
   - Create Kerberos principal for a user
   - Write file via NFS with that user's Kerberos credentials
   - Read file via SMB with same user's Kerberos credentials
   - Verify: same user identity resolved in both protocols (identity mapper consistency)
   - Verify: file content matches

For Kerberos tests, reuse the KDCHelper from `test/e2e/framework/kerberos.go`. Follow the exact patterns from `test/e2e/smb_kerberos_test.go` for KDC setup, principal creation, and keytab management.

Skip Kerberos tests if KDC container is not available (t.Skip("KDC container not available")).
