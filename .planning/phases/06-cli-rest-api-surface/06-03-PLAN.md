<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User shell → dfsctl binary | User-supplied share name is an opaque string; passed to apiclient which URL-escapes via the existing helpers |
| dfsctl → controlplane REST | Admin JWT required; Plan 02 middleware enforces |
| CLI output → user terminal | No escape concerns — share names are ASCII paths in this project's convention |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-03-01 | Input validation | CLI share name arg | mitigate | `cobra.ExactArgs(1)` ensures exactly one positional argument. The apiclient (Plan 02) URL-escapes via the existing `normalizeShareNameForAPI` helper. No shell-metacharacter expansion risk at the CLI layer (Go's flag parsing does not invoke a shell). |
| T-06-03-02 | EoP / Authz | Admin-only REST route consumption | accept | CLI just issues the HTTP call; authz enforcement lives server-side (Plan 02). A non-admin invoking `dfsctl share alice disable` gets a 403 from the server, propagated back as an error. |
| T-06-03-03 | Information Disclosure | `share show` / `share list` -o json output | accept | Share fields (including the new `Enabled`) are admin-visible by definition; no PII exposed. |
| T-06-03-04 | Destructive-op safety | `share disable` | mitigate | D-36 says disable is synchronous-drain; D-37 says brief success output. No confirmation prompt is required (per Phase 5 analysis — disable is reversible, unlike `share delete`). The CHANGELOG explicitly documents the clients-disconnect semantics. |
| T-06-03-05 | Destructive-op safety | `share enable` | accept | D-38 explicitly: no guard rails, flip the bit. Phase 5 D-04 puts the timing responsibility on the operator. |
| T-06-03-06 | Repudiation | Missing audit trail for disable/enable | accept | The REST handlers log via `internal/logger` at Info level (per Plan 02 patterns); the CLI itself does not emit audit records. This is consistent with the rest of dfsctl's admin actions (delete, edit) which also rely on server-side logs. |
</threat_model>
