<objective>
Wire DirChangeNotifier into MetadataService mutation methods (CreateFile, RemoveFile, Move, CreateDirectory, RemoveDirectory), refactor NFS4 handlers to stop calling StateManager.NotifyDirChange directly, and ensure all directory change notifications flow through the unified path: MetadataService -> LockManager -> BreakCallbacks -> {NFS4 StateManager, SMB LeaseManager}.

Purpose: Unify the directory change notification path so that ANY protocol's mutation (NFS or SMB) triggers BOTH SMB directory lease breaks AND NFS directory delegation recalls through a single dispatch point (LEASE-03, LEASE-04, ARCH-01).

Output: MetadataService wired with DirChangeNotifier, NFS4 handlers simplified (no direct NotifyDirChange), unified notification flow for cross-protocol coordination.
</objective>

    - Add field: `dirChangeNotifiers map[string]lock.DirChangeNotifier` — per-share notifier (since LockManagers are per-share). Key is shareName.
    - Add method: `SetDirChangeNotifier(shareName string, n lock.DirChangeNotifier)` — stores notifier for a share.
    - Add helper: `notifyDirChange(shareName string, parentHandle FileHandle, changeType lock.DirChangeType, ctx *AuthContext)`:
      ```go
      func (s *MetadataService) notifyDirChange(shareName string, parentHandle FileHandle, changeType lock.DirChangeType, ctx *AuthContext) {
          notifier, ok := s.dirChangeNotifiers[shareName]
          if !ok || notifier == nil {
              return
          }
          originClient := ""
          if ctx != nil && ctx.Identity != nil {
              originClient = ctx.Identity.ClientID()
          }
          // Fire-and-forget: notifier handles dispatch
          notifier.OnDirChange(lock.FileHandle(parentHandle), changeType, originClient)
      }
      ```
    - Initialize `dirChangeNotifiers` in `New()`: `dirChangeNotifiers: make(map[string]lock.DirChangeNotifier)`.

The key is to pass a string that identifies the originating client so their own leases aren't broken. Use whatever is available in the existing AuthContext struct. If Identity is not available, use ClientAddr as the originClientID.

    **Per CONTEXT.md locked decision: "NFS handlers no longer call StateManager.NotifyDirChange() directly. All directory change notifications flow through: NFS handlers -> MetadataService -> LockManager -> BreakCallbacks -> StateManager. One unified path for both protocols."**

    **1. Remove direct NotifyDirChange calls from NFS4 handlers:**

    In each of these files, find and REMOVE calls to `stateManager.NotifyDirChange(...)` or `sm.NotifyDirChange(...)`:
    - `internal/adapter/nfs/v4/handlers/create.go` — remove NotifyDirChange after file/dir creation
    - `internal/adapter/nfs/v4/handlers/remove.go` — remove NotifyDirChange after file/dir removal
    - `internal/adapter/nfs/v4/handlers/rename.go` — remove NotifyDirChange after rename
    - `internal/adapter/nfs/v4/handlers/link.go` — remove NotifyDirChange after link creation
    - `internal/adapter/nfs/v4/handlers/open.go` — remove NotifyDirChange if called on open-with-create
    - `internal/adapter/nfs/v4/handlers/setattr.go` — SPECIAL CASE: Per CONTEXT.md, "SetInfo/attribute changes do NOT break directory leases." But NFS4 currently sends NOTIFY4_CHANGE_DIR_ATTRS. Per Research recommendation, we have two options: (a) leave setattr NotifyDirChange calls as-is since they go through NFS4 StateManager for NFS4-specific attr notifications that don't correspond to SMB directory lease breaks, or (b) route through unified path with a DirChangeAttributes type that LockManager ignores for SMB but forwards to NFS4. **Choose option (a):** Keep setattr's direct NotifyDirChange call since DirChangeNotifier only has 3 types (Add/Remove/Rename) and NFS attr notifications are NFS4-specific. This is the cleanest approach.

    **2. Update NFS4 StateManager to handle directory breaks via BreakCallbacks:**

    The NFS4 StateManager already registers as BreakCallbacks (or should). In `internal/adapter/nfs/v4/state/manager.go`:
    - Ensure StateManager implements `lock.BreakCallbacks` (it may already)
    - In `OnOpLockBreak(handleKey, lock, breakToState)`: check if `lock.Lease != nil && lock.Lease.IsDirectory`. If yes, this is a directory lease break. Map it to NFS4 directory delegation recall:
      - Find delegation for the file handle
      - Send CB_RECALL to NFS4 client
      - This replaces the direct NotifyDirChange path
    - If it's a file lease break (IsDirectory=false), handle as before (delegation recall for file)

    **3. Ensure NFS4 StateManager.NotifyDirChange still exists** for:
    - NFS4 setattr notifications (kept as direct calls)
    - Any NFS4-specific notifications not covered by the 3 DirChangeTypes

    Do NOT delete NotifyDirChange entirely. Only remove the calls from create/remove/rename/link handlers. The setattr handler keeps its direct call.

    **4. Wire up in runtime/adapter initialization:**

    In the adapter/runtime wiring code (likely `pkg/adapter/smb/adapter.go` or `pkg/config/runtime.go`):
    - When creating shares and LockManagers, call `metadataService.SetDirChangeNotifier(shareName, lockManager)` to wire LockManager as the DirChangeNotifier for that share.
    - Ensure NFS4 StateManager is registered as BreakCallbacks on LockManager: `lockManager.RegisterBreakCallbacks(nfsStateManager)`.
    - Ensure SMB LeaseManager's SMBBreakHandler is also registered: `lockManager.RegisterBreakCallbacks(smbBreakHandler)`.

    **NOTE:** LockManager.RegisterBreakCallbacks currently takes a single BreakCallbacks. Check if it supports multiple registrations. If not, create a composite/multiplexer:
    ```go
    type MultiBreakCallbacks struct {
        callbacks []lock.BreakCallbacks
    }
    func (m *MultiBreakCallbacks) OnOpLockBreak(handleKey string, l *lock.UnifiedLock, breakToState uint32) {
        for _, cb := range m.callbacks {
            cb.OnOpLockBreak(handleKey, l, breakToState)
        }
    }
    // ... same for OnByteRangeRevoke, OnAccessConflict
    ```
    Place this in `pkg/metadata/lock/` as `callbacks.go` or add it to `oplock_break.go`.

    **5. Verify the full notification flow works:**
    - NFS CREATE -> MetadataService.CreateFile -> notifyDirChange -> LockManager.OnDirChange -> break SMB dir leases -> BreakCallbacks.OnOpLockBreak -> {NFS4 StateManager: recall delegation, SMB LeaseManager: send break notification}
    - SMB CREATE -> (via handler) MetadataService.CreateFile -> same flow
    - Single unified path for both protocols
