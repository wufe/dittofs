Fix 6 smbtorture ChangeNotify edge case tests:

**1. smb2.notify.valid-req (CompletionFilter validation):**
Per MS-SMB2 3.3.5.19, if CompletionFilter is 0 or contains reserved bits, the server MUST fail the request with STATUS_INVALID_PARAMETER.

Valid CompletionFilter bits per MS-SMB2 2.2.35:
- FILE_NOTIFY_CHANGE_FILE_NAME     = 0x00000001
- FILE_NOTIFY_CHANGE_DIR_NAME      = 0x00000002
- FILE_NOTIFY_CHANGE_ATTRIBUTES    = 0x00000004
- FILE_NOTIFY_CHANGE_SIZE          = 0x00000008
- FILE_NOTIFY_CHANGE_LAST_WRITE    = 0x00000010
- FILE_NOTIFY_CHANGE_LAST_ACCESS   = 0x00000020
- FILE_NOTIFY_CHANGE_CREATION      = 0x00000040
- FILE_NOTIFY_CHANGE_EA            = 0x00000080
- FILE_NOTIFY_CHANGE_SECURITY      = 0x00000100
- FILE_NOTIFY_CHANGE_STREAM_NAME   = 0x00000200
- FILE_NOTIFY_CHANGE_STREAM_SIZE   = 0x00000400

Valid mask = 0x000007FF. In the ChangeNotify handler (stub_handlers.go ChangeNotify method or change_notify.go DecodeChangeNotifyRequest), add validation:
```go
const validCompletionFilterMask = uint32(0x7FF)
if req.CompletionFilter == 0 || (req.CompletionFilter & ^validCompletionFilterMask) != 0 {
    return &HandlerResult{Status: types.StatusInvalidParameter}, nil
}
```

**2. smb2.notify.handle-permissions (FILE_LIST_DIRECTORY required):**
Per MS-SMB2 3.3.5.19, the server MUST verify that the open was granted FILE_LIST_DIRECTORY access. If not, fail with STATUS_ACCESS_DENIED.

In the ChangeNotify handler, before registering the notification, check:
```go
if ctx.DesiredAccess & FILE_LIST_DIRECTORY == 0 {
    return &HandlerResult{Status: types.StatusAccessDenied}, nil
}
```
FILE_LIST_DIRECTORY = 0x00000001 per MS-SMB2 2.2.13.1.

This requires that the ChangeNotify handler has access to the open handle's DesiredAccess. Check how SMBHandlerContext.DesiredAccess is populated — it should come from the open handle's granted access stored when CREATE completes.

**3. smb2.notify.overflow (buffer overflow notification):**
Per MS-SMB2 3.3.5.19, if the number of changes exceeds MaxOutputResponse (the OutputBufferLength field from the request), the server returns STATUS_NOTIFY_ENUM_DIR (0x0000010C) with zero output bytes. This tells the client to re-enumerate the directory.

In the NotifyRegistry delivery path (sendAndUnregister or the change accumulation logic), check if the encoded FileNotifyInformation size exceeds the PendingNotify's OutputBufferLength. If so, send STATUS_NOTIFY_ENUM_DIR instead of the change list.

The ChangeNotifyResponse already has a mechanism for this (check for OverflowOccurred or similar). If not, add:
```go
encoded := EncodeFileNotifyInformation(changes)
if len(encoded) > int(notify.OutputBufferLength) {
    // Overflow — return STATUS_NOTIFY_ENUM_DIR
    resp := &ChangeNotifyResponse{Status: types.StatusNotifyEnumDir}
    notify.AsyncCallback(notify.SessionID, notify.MessageID, notify.AsyncId, resp)
    return
}
```

Ensure STATUS_NOTIFY_ENUM_DIR constant exists in types/constants.go (value 0x0000010C). If not, add it.

**4. smb2.notify.session-reconnect:**
When a session is reconnected (durable handle reconnect), pending CHANGE_NOTIFY requests on handles belonging to that session should be preserved and continue to fire. Currently, session cleanup (UnregisterAllForSession) likely removes all pending notifications. For session RECONNECT (not logoff/disconnect), the notifications should be re-associated with the new session.

Check if the session reconnect code path (in session_setup handler or durable handle reconnect) calls UnregisterAllForSession. If so, that call should be conditional — only on actual session teardown, not on reconnect.

**5. smb2.notify.invalid-reauth:**
When a session is re-authenticated (re-auth, not reconnect), pending notifications should remain valid. Re-auth changes the session key but preserves the session state. Verify that re-auth does NOT call UnregisterAllForSession.

**6. smb2.change_notify_disabled.notfiy_disabled:**
When the server has change notifications disabled (this may be a share-level or server-level setting), CHANGE_NOTIFY should return STATUS_NOT_SUPPORTED or STATUS_NOT_IMPLEMENTED. Check if there's a configuration flag for disabling change notifications. If not, this test may not be fixable without adding a config option. If the test expects a specific error code when the handle is not a directory, verify that CHANGE_NOTIFY on a file handle returns STATUS_INVALID_PARAMETER per MS-SMB2 3.3.5.19.

Add unit tests for CompletionFilter validation and overflow behavior in change_notify_test.go.

Fix the 5 WPTS expected failures to reduce expected count to 0:

**WPTS Timestamp Tests (3):**

1. **Algorithm_NotingFileModified_Dir_LastAccessTime:**
Per MS-FSA 2.1.4.4, when a child file is modified (written to), the parent directory's LastAccessTime should be auto-updated to the current time. This is the "noting file modified" algorithm.

In the WriteFile code path (pkg/metadata/file_modify.go or the SMB write handler), after a successful write to a file, update the parent directory's LastAccessTime:
- Look up the file's parent directory handle
- If the parent directory's LastAccessTime is NOT frozen (not set to -1), update it to time.Now()

The SMB WRITE handler calls metadataService.WriteFile. After a successful write, the handler should also update the parent directory's access time. Check if this is already done in the metadata layer or needs to be added in the SMB handler layer.

2. **FileInfo_Set_FileBasicInformation_Timestamp_MinusOne_Dir_ChangeTime:**
Per MS-FSA 2.1.5.14.2, setting ChangeTime to -1 (0xFFFFFFFFFFFFFFFF in FILETIME) freezes the ChangeTime. When ChangeTime is frozen on a directory, child operations (create/delete/rename within the directory) should NOT update the directory's ChangeTime.

Check set_info.go for timestamp freeze handling. The freeze tracking was implemented in Phase 73 with per-field tracking. Verify that:
- Directory ChangeTime freeze is tracked per-handle
- When a child operation would update the parent directory's ChangeTime, it checks if ChangeTime is frozen on ANY open handle for that directory
- If frozen, skip the ChangeTime update

The issue may be that child operations (CreateFile, RemoveFile) always update the parent ChangeTime without checking the freeze state. Add a check in the metadata service or SMB handler layer.

3. **FileInfo_Set_FileBasicInformation_Timestamp_MinusTwo_Dir_LastWriteTime:**
Per MS-FSA 2.1.5.14.2, setting LastWriteTime to -2 (0xFFFFFFFFFFFFFFFE in FILETIME) unfreezes the timestamp and sets it to the current time. After unfreeze, subsequent child operations should auto-update LastWriteTime normally.

Verify that:
- Unfreeze (-2) for directory LastWriteTime works (sets to current time)
- After unfreeze, child file operations (create/write/delete) update the directory's LastWriteTime
- This requires the metadata layer to track "was this timestamp recently unfrozen" or simply rely on the normal auto-update behavior post-unfreeze

**WPTS ChangeNotify Tests (2):**

4. **BVT_SMB2Basic_ChangeNotify_ChangeSecurity:**
This test sets a security descriptor on a file and expects the parent directory's ChangeNotify watcher to fire with FILE_NOTIFY_CHANGE_SECURITY.

Check the SET_INFO handler for SecurityInformation class — after successfully setting a security descriptor, it should call NotifyRegistry.NotifyChange with the security change action. The change action constant for security is FILE_ACTION_MODIFIED_STREAM or a security-specific constant.

Actually, the CompletionFilter bit is FILE_NOTIFY_CHANGE_SECURITY (0x00000100) and the action should be FILE_ACTION_MODIFIED (0x00000003). Check that the MatchesFilter function in change_notify.go maps FILE_ACTION_MODIFIED + security-related operations to the CHANGE_SECURITY completion filter bit.

In the SET_INFO handler for SecurityInformation, add a NotifyChange call:
```go
if h.NotifyRegistry != nil {
    h.NotifyRegistry.NotifyChange(shareName, parentPath, fileName, FileActionModified)
}
```
AND ensure the notification carries metadata indicating this is a security change so MatchesFilter can check against FILE_NOTIFY_CHANGE_SECURITY.

5. **BVT_SMB2Basic_ChangeNotify_ServerReceiveSmb2Close:**
When a CLOSE command is received for a handle that has a pending ChangeNotify, the server must:
a) Cancel the pending notification
b) Send a STATUS_CANCELLED async response for the notification
c) Complete the CLOSE normally

Check the CLOSE handler to see if it calls NotifyRegistry.Unregister for the file handle being closed. If not, add it. The unregister should trigger the async callback with STATUS_CANCELLED for the pending notification.

In the CLOSE handler (stub_handlers.go or the close handler file), add:
```go
if h.NotifyRegistry != nil {
    if pending := h.NotifyRegistry.Unregister(fileID); pending != nil {
        // Send STATUS_CANCELLED for the pending notification
        resp := &handlers.ChangeNotifyResponse{Status: types.StatusCancelled}
        pending.AsyncCallback(pending.SessionID, pending.MessageID, pending.AsyncId, resp)
    }
}
```
