Fix compound request handling in compound.go to pass the following smbtorture tests. Each fix is enumerated:

**1. related3/5/6 (related request edge cases):**
These tests verify that related compound operations correctly propagate FileID from CREATE to subsequent commands (GETINFO, CLOSE, SETINFO, etc.) even with various command combinations. The current code handles basic related chaining but may fail on:
- related3: Related SETINFO after CREATE — verify ProcessRequestWithInheritedFileID correctly injects FileID into SETINFO body (the FileID at offset 8 in the SETINFO request body must be replaced with lastFileID)
- related5: Related CLOSE after failed CREATE — when a CREATE fails, subsequent related commands should get STATUS_INVALID_PARAMETER (not FILE_CLOSED), because the FileID sentinel 0xFFFFFFFFFFFFFFFF is not valid
- related6: Related GETINFO after CLOSE — after CLOSE succeeds, the file is gone; related GETINFO should get FILE_CLOSED

Audit ProcessRequestWithInheritedFileID (in response.go) to ensure it correctly replaces FileID in the request body for all command types that carry a FileID (READ, WRITE, CLOSE, QUERY_INFO, SET_INFO, FLUSH, LOCK, IOCTL). The FileID is always at body offset 8..24 (PersistentFileID 8 bytes + VolatileFileID 8 bytes) for these commands per MS-SMB2 spec.

**2. related4/7/8 (access control and CHANGE_NOTIFY in compound):**
- related4: Compound CREATE + GETINFO where CREATE uses restricted DesiredAccess. Verify that the handler context correctly propagates the open handle's DesiredAccess for permission checks in subsequent related commands.
- related7: Similar to related4 but with different access mask combinations.
- related8: Compound with CHANGE_NOTIFY as a related command. Since CHANGE_NOTIFY returns STATUS_PENDING (async), the compound must send an interim response for that command and continue processing subsequent commands.

For related8: When a command returns STATUS_PENDING with an AsyncId, the compound must include an interim response (status=STATUS_PENDING, flags=SMB2_FLAGS_ASYNC_COMMAND) at that position and continue processing remaining commands. The actual async completion is sent separately later.

**3. unrelated1 (unrelated request handling):**
Unrelated compound requests do NOT inherit FileID/SessionID/TreeID from previous commands. Each command uses its own values. The current code already handles this, but verify:
- When hdr.IsRelated() is false, FileID is extracted from the command body via ExtractFileID
- SessionID and TreeID come from the command's own header, not inherited

**4. invalid1/2/3/4 (parameter validation):**
- invalid1: Compound with invalid SMB2 magic in second command — should return STATUS_INVALID_PARAMETER for that command and all subsequent related commands
- invalid2: Compound with invalid structure size — similar to invalid1
- invalid3: Compound where a command references a closed file (FILE_CLOSED detection) — verify that CLOSE followed by related READ returns STATUS_FILE_CLOSED
- invalid4: Compound with invalid NextCommand alignment — if NextCommand is not 8-byte aligned, return STATUS_INVALID_PARAMETER

Add validation in ParseCompoundCommand: if NextCommand is non-zero and not 8-byte aligned, return an error. Currently the function trusts NextCommand blindly.

**5. compound-break (oplock break interaction):**
When a compound operation triggers an oplock/lease break on another client, the break notification must be sent asynchronously without blocking the compound processing. Verify that the current oplock break delivery (via lease/notifier.go) does not block the compound response path. If it does, ensure break notifications are dispatched via goroutine.

**6. create-write-close (CREATE+WRITE+CLOSE sequence):**
This is a common compound pattern. Verify that:
- CREATE returns FileID in response
- WRITE uses the FileID from CREATE (via related operations)
- CLOSE uses the FileID from the WRITE (or CREATE)
- All three commands appear in a single compound response frame
- The file is visible after the compound completes

**7. compound-padding (8-byte response padding):**
The current sendCompoundResponses already pads to 8-byte alignment. Verify that the padding applies to ALL commands including error responses. Also verify that NextCommand offsets account for padding.

**8. compound_find.related/unrelated/close:**
These test QUERY_DIRECTORY (find) operations in compound requests:
- compound_find_related: CREATE dir + QUERY_DIRECTORY (related) should work
- compound_find_unrelated: CREATE dir + QUERY_DIRECTORY (unrelated, own handle) should work
- compound_find_close: CREATE dir + QUERY_DIRECTORY + CLOSE in compound

Ensure ExtractFileID handles SMB2_QUERY_DIRECTORY command (the FileID is at body offset 8..24 for QUERY_DIRECTORY too per MS-SMB2 2.2.33).

**9. interim1/2/3 (STATUS_PENDING interim responses in compound):**
When a command in a compound returns STATUS_PENDING (async), the server must:
a) Include an interim response at that command's position in the compound
b) Set SMB2_FLAGS_ASYNC_COMMAND flag on the interim response
c) Set the AsyncId in the interim response header
d) Continue processing subsequent commands in the compound
e) Send the actual async completion as a separate standalone response later

Modify ProcessCompoundRequest to detect when a command result has Status == StatusPending and AsyncId != 0, and emit an interim response entry in the compound. The actual completion must be handled by the async callback (already wired for CHANGE_NOTIFY).

For non-CHANGE_NOTIFY async operations (interim1/2/3 test general async), implement a generic async dispatch mechanism in the compound path that:
- Detects STATUS_PENDING from any handler
- Records the pending async operation
- Includes interim response in compound frame
- Sends completion separately via async callback

Reference the AsyncResponseRegistry pattern from change_notify.go for this.
