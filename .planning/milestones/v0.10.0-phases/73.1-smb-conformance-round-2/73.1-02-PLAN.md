Fix three specific CREATE edge cases:

**1. Leading slash path handling (smb2.create.leading-slash):**
Per MS-SMB2 2.2.13, the Buffer field in CREATE request contains the filename relative to the share root. Windows clients may send paths with a leading backslash (e.g., `\foo\bar`). The current code at line 416-417 does:
```go
filename := strings.ReplaceAll(req.FileName, "\\", "/")
filename = strings.TrimPrefix(filename, "/")
```
This trims one leading slash but the smbtorture test sends `\\foo` (double backslash which becomes `//foo` after replace). Add handling for multiple leading slashes:
```go
filename = strings.TrimLeft(filename, "/")
```
Use TrimLeft instead of TrimPrefix to handle multiple leading separators.

Also handle the edge case where the entire path is just slashes — this should resolve to the share root (empty filename after trim), which is already handled by the root directory code path.

**2. MKDIR visibility (smb2.create.mkdir-visible):**
When CREATE with FILE_DIRECTORY_FILE creates a new directory, the directory must be immediately visible to other clients doing QUERY_DIRECTORY on the parent. This is typically a timing issue. Check that:
- CreateDirectory in metadata service flushes to the store before returning
- The ChangeNotify trigger fires AFTER the directory is committed (not before)
- There is no caching layer that delays visibility

Read pkg/metadata/file_create.go to verify CreateDirectory commits atomically. If there's a write-behind or lazy flush, ensure it's synchronous for directory creates.

**3. Create context blob validation (smb2.create.blob, smb2.durable-v2-open.create-blob):**
Per MS-SMB2 2.2.13.2, create context entries have a specific layout:
- NameOffset (2 bytes), NameLength (2 bytes), Reserved (2 bytes)
- DataOffset (2 bytes), DataLength (4 bytes)
- Next (4 bytes) — offset to next entry, 0 for last
- Name and Data follow at their respective offsets

The smbtorture blob test sends intentionally malformed create context blobs (bad offsets, overlapping entries, missing data) and expects STATUS_INVALID_PARAMETER.

Add validation in the create context parsing code (likely in create.go or encoding.go where create contexts are decoded):
- Verify NameOffset >= 16 (header size) and NameOffset + NameLength <= total buffer
- Verify DataOffset >= 16 and DataOffset + DataLength <= total buffer
- Verify Next is 0 (last) or 8-byte aligned and points within the buffer
- On any validation failure, return STATUS_INVALID_PARAMETER

Add unit tests for all three fixes in create_test.go:
- TestCreate_LeadingSlashNormalization: verify `\\foo`, `////bar`, `/` all normalize correctly
- TestCreate_CreateContextBlobValidation: verify malformed context buffers return error

Fix ADS/streams handling to pass smbtorture streams tests. ADS (Alternate Data Streams) are stored as directory children with names like `file:stream:$DATA` per the existing architecture from Phase 72-73.

**1. streams.attributes1/attributes2 (ADS attributes):**
Per MS-FSA 2.1.5.11.22 (FileStreamInformation), when querying stream attributes:
- attributes1: FILE_ATTRIBUTE_ARCHIVE should be returned for the default data stream
- attributes2: Stream size and allocation size must reflect the actual stream content
Verify that FileStreamInformation in query_info.go returns correct StreamSize (actual bytes written) and StreamAllocationSize (rounded up to block boundary, typically 4096-byte aligned). If allocation size is returned as 0 for non-empty streams, fix it.

**2. streams.create-disposition (ADS create disposition):**
When CREATE targets a named stream (e.g., `file:stream:$DATA`), the create disposition (OPEN, CREATE, OPEN_IF, OVERWRITE, OVERWRITE_IF, SUPERSEDE) must work the same as for regular files, but applied to the stream, not the base file. Verify that create.go handles dispositions correctly when the filename contains a colon-separated stream name.

**3. streams.delete (ADS delete):**
DELETE_ON_CLOSE on a named stream should delete only that stream, not the base file. Verify that the CLOSE handler checks if the open handle points to a stream and only deletes the stream entry, leaving the base file intact.

**4. streams.io (ADS I/O):**
READ and WRITE on named streams must use the stream's own PayloadID (block store content address), not the base file's. Verify that GetBlockStoreForHandle resolves the stream's metadata entry correctly.

**5. streams.names/names2/names3 (ADS name enumeration):**
FileStreamInformation must enumerate ALL streams including the default data stream (`::$DATA`). The default stream should always be listed first with the name `::$DATA`. Additional named streams should follow with names like `:streamname:$DATA`. Verify the enumeration includes the default stream and all named streams.

**6. streams.rename2 (ADS rename):**
Renaming a stream (via SET_INFO FileRenameInformation) should change the stream name within the base file's namespace. Verify set_info.go handles stream rename (the new name in FileRenameInformation may be a stream name like `:newname:$DATA`).

**7. streams.sharemodes (ADS share mode enforcement):**
Per MS-FSA 2.1.5.1.2, share mode enforcement spans the base file and all its streams. If a stream is opened with SHARE_READ, opening the base file or another stream must respect that share mode. Verify that checkShareModeConflict (or equivalent) considers all opens on the same base file, including streams.

**8. streams.dir (ADS directory listing):**
When a directory is queried (QUERY_DIRECTORY), files with ADS should NOT list the streams as separate directory entries. Only the base file appears. Verify that query_directory.go filters out stream entries from directory listings.

**9. streams.basefile-rename-with-open-stream:**
Renaming a base file that has open streams should either succeed (if share modes allow) or fail with STATUS_SHARING_VIOLATION. Verify the rename handler checks for open stream handles.

**10. streams.zero-byte (ADS zero-byte handling):**
A zero-byte stream should be creatable and queryable. FileStreamInformation should show StreamSize=0 for empty streams.

**11. create_no_streams.no_stream:**
When FILE_NO_EA_KNOWLEDGE is set in CreateOptions, or when the test checks that stream-unaware opens work, verify that opening a file with `::$DATA` suffix still resolves to the base file.
