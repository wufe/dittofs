Build on the interim response infrastructure from Plan 01 to implement full compound async dispatch. The smbtorture compound_async tests verify that when an operation within a compound would block (e.g., flush, large read/write, rename with contention), the server can:

1. Send the compound response with an interim STATUS_PENDING for the slow operation
2. Complete the slow operation asynchronously
3. Send the completion as a separate standalone response with the matching AsyncId

**Architecture:**

The compound_async tests fall into two categories:

**Category A — Operations that ARE async (flush_close, flush_flush, read_read, write_write, getinfo_middle, create_lease_break_async):**
These test that certain operations CAN be made async within a compound. Per MS-SMB2 3.3.5.2.7, the server MAY process compound commands asynchronously by returning STATUS_PENDING for any command and completing it later.

However, the smbtorture tests may actually be testing that the server handles compounds WHERE the test client expects async behavior. Looking at the smbtorture source, these tests typically:
1. Send a compound where one operation might trigger async (e.g., FLUSH which may need to wait for data sync)
2. Expect either synchronous completion OR interim + async completion
3. Verify the compound response structure is correct in either case

**Category B — Non-async operations (rename_non_compound_no_async, rename_same_srcdst_non_compound_no_async):**
These test that a rename operation that does NOT need async completes synchronously in a single response (no STATUS_PENDING).

**Implementation approach:**

Rather than making all operations genuinely async (which would require threading all handlers through goroutines), implement the compound async framework that:

1. **Detects async-eligible operations**: FLUSH, large READ/WRITE, RENAME with potential contention
2. **For compounds where a handler returns STATUS_PENDING**: Include interim response in compound frame (already from Plan 01), send async completion later
3. **For compounds where all handlers complete synchronously**: Return normal compound response (no async)

The key missing piece is a general-purpose async completion sender (not just for CHANGE_NOTIFY). Create `SendAsyncCompletionResponse` in response.go that:
- Builds a standalone response header with FlagAsync set
- Sets the AsyncId to match the interim response
- Encodes the handler's response body
- Signs and optionally encrypts the response
- Sends it on the connection

```go
func SendAsyncCompletionResponse(sessionID, messageID, asyncId uint64, command types.Command, result *HandlerResult, connInfo *ConnInfo) error {
    respHeader := &header.SMB2Header{
        ProtocolID: header.ProtocolID,
        Command:    command,
        Status:     result.Status,
        Flags:      types.FlagResponse | types.FlagAsync,
        MessageID:  messageID,
        AsyncId:    asyncId,
        SessionID:  sessionID,
        Credits:    1,
    }
    // Encode body from result
    body := result.Body
    if body == nil {
        body = MakeErrorBody()
    }
    return SendMessage(respHeader, body, connInfo)
}
```

Then in ProcessCompoundRequest, when a handler result has AsyncId != 0 and Status == StatusPending:
- Record the pending operation with a completion callback
- Include interim response in compound
- When the operation completes (via the handler's async mechanism), call SendAsyncCompletionResponse

For FLUSH specifically: The flush handler currently completes synchronously. For the compound_async.flush_* tests, the server needs to support async flush. If the flush is fast (memory store), it completes synchronously. But for the test framework, the compound handling must be correct regardless.

The smbtorture tests primarily verify:
1. The compound response frame structure is valid (padding, NextCommand offsets)
2. Error propagation works (failed async doesn't corrupt subsequent commands)
3. FileID inheritance works across async boundaries
4. The async completion response has correct AsyncId

**For the 10 specific tests:**

- `flush_close`: Compound FLUSH + CLOSE. FLUSH may be async; CLOSE uses inherited FileID.
- `flush_flush`: Two FLUSHes in compound. Both may be async.
- `read_read`: Two READs in compound. Both complete synchronously but verify compound structure.
- `write_write`: Two WRITEs in compound. Both complete synchronously but verify compound structure.
- `getinfo_middle`: GETINFO as middle command in compound. Verify response at middle position.
- `rename_last`: RENAME as last command. Verify compound completes correctly.
- `rename_middle`: RENAME as middle command. Verify subsequent commands still work.
- `rename_non_compound_no_async`: Single RENAME (not compound). Verify it does NOT return async.
- `rename_same_srcdst_non_compound_no_async`: Rename to same name. Verify no async.
- `create_lease_break_async`: CREATE that triggers lease break during compound. The break must be sent async without blocking compound.

Implement the SendAsyncCompletionResponse function and wire it into the compound processing path. Ensure that non-async operations (like simple rename) do NOT get STATUS_PENDING — they complete inline.
