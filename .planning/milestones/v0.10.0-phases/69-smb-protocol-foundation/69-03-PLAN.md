<objective>
Wire credit validation and sequence window enforcement into the SMB request processing pipeline, and implement compound-level credit accounting.

Purpose: The data structures from Plan 02 (CommandSequenceWindow, credit validation helpers) need to be integrated into the actual request flow. This plan adds the SequenceWindow to ConnInfo, inserts validation before dispatch in ProcessSingleRequest and ProcessCompoundRequest, ensures every response (including errors) expands the sequence window, and implements compound-specific credit accounting (first command charges, middle grants 0, last grants credits).

Output: Fully enforced credit flow control for all SMB clients.
</objective>

```go
// Credit validation: per MS-SMB2 3.3.5.2.3 and 3.3.5.2.5
if !session.IsCreditExempt(reqHeader.Command, reqHeader.SessionID) {
    // Validate CreditCharge against payload size (MS-SMB2 3.3.5.2.5)
    if connInfo.SupportsMultiCredit {
        if err := session.ValidateCreditCharge(reqHeader.Command, reqHeader.CreditCharge, body); err != nil {
            logger.Debug("Credit charge validation failed",
                "command", reqHeader.Command.String(),
                "creditCharge", reqHeader.CreditCharge,
                "error", err)
            return SendErrorResponse(reqHeader, types.StatusInvalidParameter, connInfo)
        }
    }

    // Validate and consume sequence numbers from window (MS-SMB2 3.3.5.2.3)
    charge := session.EffectiveCreditCharge(reqHeader.CreditCharge)
    if !connInfo.SequenceWindow.Consume(reqHeader.MessageID, charge) {
        logger.Debug("Sequence window validation failed",
            "command", reqHeader.Command.String(),
            "messageID", reqHeader.MessageID,
            "creditCharge", charge)
        return SendErrorResponse(reqHeader, types.StatusInvalidParameter, connInfo)
    }
}
```

This goes AFTER the `WriteNetBIOSFrame` call at the bottom of `SendMessage`. This ensures every response — success, error, async — expands the window.

**5. Also update buildResponseHeaderAndBody:**

The credit grant from `GrantCredits()` is already stored in `respHeader.Credits`. The sequence window expansion reads `hdr.Credits` in `SendMessage`. So the flow is:
1. `GrantCredits()` calculates credits to grant -> stored in response header
2. `SendMessage()` sends the frame
3. `SendMessage()` calls `SequenceWindow.Grant(hdr.Credits)` to expand window

This is correct because the client only gets the credits when the response is actually sent.

Since NEGOTIATE is credit-exempt (no consumption), but the response still grants credits (which `SendMessage` will expand into the window), this flow works naturally. Verify that the NEGOTIATE response path goes through `SendMessage`.

**2. Enforce middle responses grant 0 credits, last response grants credits:**

In `sendCompoundResponses`, modify the credit granting logic:

For compound responses (len > 1):
- Before building/sending, iterate through responses:

if connInfo.SequenceWindow != nil {

Note: Check whether `sendCompoundResponses` for len==1 goes through `SendMessage` (which already has Grant). If so, the compound-specific Grant should only apply for len > 1. Read the code carefully to avoid double-granting.
