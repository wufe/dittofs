    **Session Re-Authentication (D-19, 5 tests: reauth2-6):**
    Read smbtorture session re-auth tests from Samba source (source4/torture/smb2/session.c).

    1. Per MS-SMB2 3.3.5.5, when a SESSION_SETUP arrives with an existing SessionId:
       - This is re-authentication, not a new session
       - The server MUST re-authenticate the user (run full SPNEGO/NTLM handshake)
       - On success: update the session's signing key (and encryption keys if applicable) to new derived keys
       - The session's tree connects, open files, and pending operations MUST be preserved
       - The session's SecurityContext (user identity) MAY change if a different user authenticates

    2. In session_setup.go, find where SESSION_SETUP is handled. Add re-auth detection:
       - If reqHeader.SessionID != 0 AND a session with that ID exists AND the session is fully established (not mid-handshake):
         - This is re-auth, not initial auth or session binding
         - Process the auth handshake (SPNEGO/NTLM) as normal
         - On success, update the existing session's keys using the new SessionBaseKey
         - Re-derive SigningKey, EncryptionKey, DecryptionKey per MS-SMB2 3.3.5.5.3
         - Do NOT destroy tree connects, open files, or any other session state
         - Return STATUS_SUCCESS (not MORE_PROCESSING_REQUIRED after final token)

    4. smb2.session.reauth3: Re-auth then perform file operations. Tree connects and open files work.

    6. smb2.session.reauth5: Re-auth with encryption. New encryption keys take effect.
    7. smb2.session.reauth6: Multiple re-auths in sequence. Each updates keys correctly.
