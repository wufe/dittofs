    **DH V2 Reopen (D-11, ~14 tests):**

    Per MS-SMB2 3.3.5.9.12, DH2C (Durable Handle V2 Reconnect) processing:

    1. smb2.durable-v2-open.reopen1/reopen1a: Basic DH2 reconnect after disconnect.
       - Client sends CREATE with DH2C context containing CreateGuid matching the original DH2Q
       - Server must: find persisted handle by CreateGuid, verify session key hash matches, restore the open file state (FileID, TreeConnect, lease state)
       - Fix: In durable_context.go, ensure DH2C processing:
         a) Looks up persisted handle by CreateGuid (not by path or FileID)
         b) Verifies the new session's key hash against the stored hash
         c) Generates a NEW volatile FileID (do not reuse the old one — per MS-SMB2 3.3.5.9.12 the volatile part is regenerated)
         d) Restores the open file entry in Handler.files map with the new FileID
         e) Returns the new FileID in the CREATE response
         f) Removes the persisted handle from DurableStore after successful reconnect

    2. smb2.durable-v2-open.reopen2/2b/2c: DH2 reconnect with additional validation:
       - reopen2: Reconnect after server-initiated disconnect (simulate server closing connection)
       - reopen2b: Reconnect with different tree connect path (should succeed if share is same)
       - reopen2c: Reconnect with different filename in CREATE (should fail — filename must match)
       - Fix: Add filename validation in DH2C processing. The path in the CREATE request must match the persisted handle's path.

    3. smb2.durable-v2-open.reopen1a-lease / reopen2-lease / reopen2-lease-v2: DH2 reconnect with lease.
       - After reconnect, the lease state from before disconnect must be restored
       - The RqLs context in the reconnect CREATE must be compatible with the persisted lease state
       - Fix: In DH2C processing, after restoring the open file, also restore the lease state in LeaseManager using the persisted LeaseState value

    4. smb2.durable-v2-open.create-blob: DH2Q create context validation.
       - Verify that the server correctly parses the DH2Q blob (Timeout, Flags, Reserved, CreateGuid)
       - Fix: Ensure DH2Q parsing is correct per MS-SMB2 2.2.13.2.11

    5. smb2.durable-v2-open.open-oplock / open-lease: DH2 initial open with oplock/lease.
       - Creating a file with DH2Q context and requesting an oplock or lease
       - The server must mark the handle as durable and store the CreateGuid
       - Fix: Ensure IsDurable=true is set when DH2Q is present in CREATE

    6. smb2.durable-v2-open.lock-oplock / lock-lease / lock-noW-lease: DH2 with locks across reconnect.
       - Byte-range locks must survive reconnect
       - Fix: Include lock state in persisted durable handle and restore on reconnect

    7. smb2.durable-v2-open.durable-v2-setinfo: SetInfo on reconnected DH2 handle.
       - After reconnect, SetInfo operations must work on the restored handle
       - Fix: Ensure the restored OpenFile has all necessary fields (MetadataHandle, ShareName, etc.)

    **Lease State Machine (D-12, ~9 tests):**

    Per MS-SMB2 3.3.1.4, lease state transitions:

    1. smb2.lease.request: Basic lease request on CREATE.
       - Client requests RWH lease. Server grants based on share mode and conflicts.
       - Fix: Verify lease_context.go correctly processes RqLs create context and calls LeaseManager.GrantLease with the correct state. Ensure the response includes RqLs with granted state.

    2. smb2.lease.nobreakself: Opening a second handle with the same lease key should NOT trigger a lease break to the first handle.
       - Per MS-SMB2: "If Open.Lease.LeaseKey == the new open's LeaseKey, no break is required"
       - Fix: In lease_manager.go BreakLease or the conflict check, skip break when the conflicting open has the same LeaseKey.

    3. smb2.lease.statopen / statopen4: Opening a file with NO lease request (stat-only open: FILE_READ_ATTRIBUTES access, FILE_SHARE_READ|WRITE|DELETE) should NOT break existing leases.
       - Per MS-SMB2 3.3.5.9.8: stat opens that don't request a lease and only request read-attributes access don't conflict with existing leases.
       - Fix: In the lease conflict check, skip lease break when the new open's DesiredAccess is only FILE_READ_ATTRIBUTES (0x00000080).

    4. smb2.lease.upgrade / upgrade2 / upgrade3: Lease upgrade from R to RW or RH to RWH.
       - Client sends CREATE with same LeaseKey but higher lease state.
       - Per MS-SMB2 3.3.5.9.8: "If Lease.LeaseState does not include the requested state, and no other opens conflict, grant the upgrade."
       - Fix: In lease_context.go, when processing RqLs for an existing lease key:
         a) If requested state is a superset of current state, attempt upgrade
         b) Check for conflicts with OTHER lease keys (not same key)
         c) If no conflicts, grant the upgrade
         d) Return the new lease state in the response

    **Lease Breaking States (D-12, 6 tests: breaking1-6):**

    Per MS-SMB2 3.3.1.4 breaking state:

    5. smb2.lease.breaking1-6: When a lease break is in progress (sent to client, awaiting acknowledgment), the lease is in "breaking" state.
       - breaking1: New open during breaking state. The new open should wait for (or not need to trigger) the break since it's already in progress.
       - breaking2: Lease break acknowledgment with correct break-to state.
       - breaking3: Lease break acknowledgment with lower state than requested (client can break to a lower state than what server requested).
       - breaking4: Multiple opens during breaking state.
       - breaking5: Break timeout (client doesn't acknowledge within timeout). Server should force-break to None.
       - breaking6: Break during breaking (cascading breaks).
       - Fix: In lease_manager.go:
         a) Track "breaking" state per lease (isBreaking flag + breakToState)
         b) When a break is sent, set isBreaking=true
         c) When a new conflicting open arrives during breaking, don't send another break — let the existing break complete or queue the new open to wait
         d) On acknowledgment: validate break-to state >= requested break-to, update lease state, clear isBreaking, wake queued opens
         e) On timeout: force lease state to None, clear isBreaking, wake queued opens

    **Lease V2 Epoch (D-13, 3 tests):**

    6. smb2.lease.v2_request / v2_epoch1 / v2_epoch2 / v2_epoch3: Lease V2 epoch tracking per MS-SMB2 3.3.5.2.11.
       - Each lease state change increments the epoch counter
       - Client must include the current epoch in lease acknowledgment
       - Fix: In lease_manager.go, add epoch counter to lease state. Increment on every state change. Include epoch in lease break notification. Validate epoch in acknowledgment.
