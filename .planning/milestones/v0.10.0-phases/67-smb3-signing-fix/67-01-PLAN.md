Perform a systematic spec-compliance audit of the preauth hash chain implementation. This task is diagnostic -- it identifies bugs but does NOT fix them (fixes are in Plan 02).

**Step 1: Trace the message flow end-to-end**

Read and trace the complete preauth hash lifecycle:
1. `ReadRequest` in `framing.go` strips NetBIOS framing via `readNetBIOSPayload`
2. `connection.go` reconstructs `rawMessage = hdr.Encode() + body` (line 191-193)
3. `ProcessSingleRequest` calls `RunBeforeHooks(connInfo, reqHeader.Command, rawMessage)` -- this is where NEGOTIATE request hash happens
4. Handler executes (negotiate.go or session_setup.go)
5. `SendResponseWithHooks` builds response, calls `SendMessage` (which signs), syncs signature back to header, then builds `rawResponse = respHeader.Encode() + body`, then calls `RunAfterHooks` -- this is where NEGOTIATE response hash happens
6. For SESSION_SETUP: before-hook stashes or updates per-session hash, handler calls `InitSessionPreauthHash` (for SessionID=0), `configureSessionSigningWithKey` calls `GetSessionPreauthHash` then `DeleteSessionPreauthHash`, after-hook updates per-session hash with response

**Step 2: Check each of the 4 pitfalls from issue #252**

For each pitfall, verify by reading the code:

1. **NetBIOS prefix in hash**: Verify `rawMessage` starts with SMB2 protocol ID (0xFE534D42), not NetBIOS header (0x00). Check `readNetBIOSPayload` strips correctly and `connection.go` reconstruction doesn't re-add it.

2. **Final SESSION_SETUP response in derivation hash**: Verify the ordering in `session_setup.go`:
   - `configureSessionSigningWithKey` calls `GetSessionPreauthHash(sess.SessionID)` to get the derivation hash
   - Then calls `DeleteSessionPreauthHash(sess.SessionID)` to clean up
   - The after-hook `sessionPreauthAfterHook` calls `UpdateSessionPreauthHash(sessionID, rawMessage)` -- but if the session was deleted, this is a no-op
   - Verify this ordering is correct: derivation hash is captured BEFORE the final response is hashed

3. **rawMessage roundtrip fidelity**: Check `header.Parse()` and `header.Encode()` for perfect roundtrip. Read `internal/adapter/smb/header/` package. Look for any fields that might not roundtrip (reserved fields, padding, NextCommand for non-compound).

4. **Response signature in after-hook**: For the intermediate SESSION_SETUP response (STATUS_MORE_PROCESSING_REQUIRED), verify the after-hook receives the unsigned bytes (no session key yet, so no signing). For the final SESSION_SETUP response (STATUS_SUCCESS), verify the after-hook runs AFTER `SendMessage` which does signing AND syncs the signature back to `respHeader.Signature`. The `rawResponse = respHeader.Encode() + body` should include the real signature. But WAIT -- the final response should NOT be in the derivation hash, so the after-hook's update is a no-op (session preauth already deleted). Verify this is correct.

**Step 3: Investigate WPTS BVT_Negotiate_SMB311 failures**

The 5 WPTS tests that fail are:
- BVT_Negotiate_SMB311
- BVT_Negotiate_SMB311_Preauthentication_Encryption_AES_256_CCM
- BVT_Negotiate_SMB311_Preauthentication_Encryption_AES_256_GCM
- BVT_Negotiate_SMB311_Preauthentication_Encryption_CCM
- BVT_Negotiate_SMB311_Preauthentication_Encryption_GCM

These are Negotiate-category tests. Read the WPTS test names carefully:
- The base test (`BVT_Negotiate_SMB311`) tests basic 3.1.1 negotiation
- The `_Preauthentication_Encryption_*` tests test negotiate with specific encryption cipher contexts

Likely failure causes (investigate in order):
a) Preauth hash mismatch causing signing failure on SESSION_SETUP response
b) Missing or malformed negotiate contexts in the response
c) Incorrect negotiate context alignment/padding
d) Wrong cipher negotiation behavior
