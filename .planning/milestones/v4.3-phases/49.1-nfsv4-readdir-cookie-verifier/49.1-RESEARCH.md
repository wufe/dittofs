# Phase 49.1: NFSv4 READDIR Cookie Verifier - Research

**Researched:** 2026-03-12
**Domain:** NFSv4 READDIR cookie verifier implementation (RFC 7530 Section 16.24)
**Confidence:** HIGH

## Summary

This phase implements a proper cookie verifier for the NFSv4 READDIR handler, replacing the current hardcoded 8 zero bytes. The NFSv3 implementation already has a working verifier using `directoryMtimeVerifier(mtime)` in `internal/adapter/nfs/v3/handlers/verf.go` with advisory-only mismatch validation in `readdir.go`. The NFSv4 handler needs the same pattern applied.

The change is small and well-defined: (1) fetch directory mtime before encoding the response, (2) encode `uint64(mtime.UnixNano())` as the 8-byte cookie verifier, (3) validate incoming verifier on non-initial requests with debug-level logging on mismatch (no error returned). The pseudo-fs path correctly keeps zero verifier and needs no changes.

**Primary recommendation:** Mirror the NFSv3 `directoryMtimeVerifier()` pattern exactly. Create a shared or local helper, fetch directory attributes via `metaSvc.GetFile()`, encode mtime as the verifier, and validate incoming verifiers advisory-only. This is a focused change touching ~30 lines in one handler file plus test updates.

## Architecture Patterns

### Recommended Change Structure
```
internal/adapter/nfs/v4/handlers/
  readdir.go          # Primary change: verifier generation + validation
  (no new files needed)
```

### Pattern 1: Mtime-Based Cookie Verifier (from NFSv3)
**What:** Generate an 8-byte verifier from directory modification time
**When to use:** Every real-FS READDIR response
**Example:**
```go
// Source: internal/adapter/nfs/v3/handlers/verf.go (existing NFSv3 pattern)
func directoryMtimeVerifier(mtime time.Time) uint64 {
    return uint64(mtime.UnixNano())
}
```

### Pattern 2: Advisory Verifier Validation (from NFSv3)
**What:** Check incoming verifier, log mismatch, but continue serving entries
**When to use:** Non-initial READDIR requests (cookie != 0, verifier != 0)
**Example:**
```go
// Source: internal/adapter/nfs/v3/handlers/readdir.go:163-178 (existing pattern)
if cookie != 0 && cookieVerf != 0 && cookieVerf != currentVerifier {
    logger.Debug("NFSv4 READDIR: directory modified since last read, continuing with current entries",
        "handle", fmt.Sprintf("%x", ctx.CurrentFH),
        "expected_verf", fmt.Sprintf("0x%016x", cookieVerf),
        "current_verf", fmt.Sprintf("0x%016x", currentVerifier),
        "client", ctx.ClientAddr)
}
```

### Pattern 3: Fetching Directory Attributes for Verifier
**What:** Use `MetadataService.GetFile()` to obtain directory mtime
**When to use:** Before encoding READDIR response
**Example:**
```go
// Source: internal/adapter/nfs/v4/handlers/getattr.go:83 (established pattern)
// In readDirRealFS, after getting metaSvc:
dirFile, err := metaSvc.GetFile(authCtx.Context, metadata.FileHandle(ctx.CurrentFH))
if err != nil {
    // handle error
}
currentVerifier := uint64(dirFile.Mtime.UnixNano())
```

### Pattern 4: Encoding Verifier as 8 Bytes
**What:** Write uint64 verifier as 8 big-endian bytes (NFSv4 verifier4 = opaque[8])
**When to use:** In response buffer construction
**Example:**
```go
// Replace: buf.Write(make([]byte, 8))
// With:
var verfBytes [8]byte
binary.BigEndian.PutUint64(verfBytes[:], currentVerifier)
buf.Write(verfBytes[:])
```

### Anti-Patterns to Avoid
- **Returning NFS4ERR_NOT_SAME on mismatch:** Breaks macOS Finder folder copy with error -8062. The RFC allows it, but Linux knfsd and our NFSv3 implementation deliberately avoid it.
- **Using ctime for verifier:** FATTR4_CHANGE uses ctime, but the verifier should track directory content changes (mtime), not metadata changes.
- **Validating verifier for pseudo-fs:** Pseudo-fs is ephemeral and rarely changes; zero verifier is correct.
- **Creating a separate verifier file:** The helper is simple enough to be a local function or package-level function in `readdir.go`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XDR uint64 encoding | Manual byte manipulation | `encoding/binary.BigEndian.PutUint64` | Consistent with `serverBootVerifier` pattern in `write.go` |
| Directory attribute fetch | Direct store access | `metaSvc.GetFile()` | Handles share routing, pending writes, caching |

**Key insight:** This is a straightforward port of the NFSv3 pattern. No new abstractions needed.

## Common Pitfalls

### Pitfall 1: Extra GetFile Call Performance
**What goes wrong:** Adding a `GetFile()` call to fetch directory mtime adds an extra metadata store round-trip.
**Why it happens:** The current `readDirRealFS` doesn't fetch the directory's own attributes.
**How to avoid:** The `ReadDirectory()` call already fetches the directory internally for type/permission checking. The additional `GetFile()` for the verifier is a second lookup, but for memory/BadgerDB stores this is negligible (~microseconds). If needed later, `ReadDirectory` could return directory mtime, but that's a refactor out of scope.
**Warning signs:** Only relevant if profiling shows READDIR as a bottleneck, which is unlikely given current use.

### Pitfall 2: Decoding Incoming Verifier as [8]byte vs uint64
**What goes wrong:** The current code reads the incoming verifier as `[8]byte` but needs to compare it as `uint64` against the computed mtime verifier.
**Why it happens:** The NFSv4 wire format is opaque[8], not uint64. Need to convert.
**How to avoid:** Decode the 8-byte buffer as `uint64` using `binary.BigEndian.Uint64(cookieVerf[:])` for comparison.
**Warning signs:** Byte-level comparison would also work but is less clear.

### Pitfall 3: Zero Mtime Edge Case
**What goes wrong:** If directory mtime is zero (time.Time zero value), the verifier would be 0, making it indistinguishable from "no verifier."
**Why it happens:** Unlikely in practice since all store implementations set mtime on creation.
**How to avoid:** This is acceptable behavior -- if mtime is zero, the verifier is effectively disabled (same as current behavior). No special handling needed.

### Pitfall 4: Test Helper Sends Zero Verifier
**What goes wrong:** The `encodeReadDir()` test helper in `ops_test.go` always sends 8 zero bytes for the verifier, which means it never triggers the validation path.
**Why it happens:** Tests were written when verifier was always zero.
**How to avoid:** Update `encodeReadDir()` to optionally accept a verifier parameter, or create a new `encodeReadDirWithVerf()` helper. Add dedicated tests for verifier mismatch path.

## Open Questions

1. **Should `readDirRealFS` reuse the `authCtx` from the existing `buildV4AuthContext` call for `GetFile()`?**
   - What we know: `readDirRealFS` already calls `buildV4AuthContext` at line 93 and uses the result for `ReadDirectory`. `GetFile` needs a `context.Context`, not `AuthContext`.
   - What's unclear: Whether to call `GetFile` with `authCtx.Context` (auth-enriched) or `ctx.Context` (compound context). Both should work since `GetFile` doesn't check permissions.
   - Recommendation: Use `authCtx.Context` for consistency, since it carries the correct share routing.

2. **Should `GetFile` be called before or after `ReadDirectory`?**
   - What we know: We need mtime for both the verifier response AND for incoming verifier validation.
   - What's unclear: If we validate the incoming verifier before `ReadDirectory`, we need `GetFile` first. If we only use it for the response, it could come after.
   - Recommendation: Call `GetFile` before `ReadDirectory` so we can validate the incoming verifier early and log before the directory scan.
